Add local-first application and AI observability for reasoning workflows.

Context:
The PKE now executes LLM-based evidence reasoning through Ollama. Reasoning can be slow, but the system does not expose enough telemetry to distinguish prompt construction, retrieval, inference, validation and persistence latency.

Goal:
Instrument reasoning workflows and provide an optional local observability stack for application telemetry and LLM-specific traces.

Architecture:
PKE
→ OpenTelemetry SDK
→ OpenTelemetry Collector
→ VictoriaMetrics for metrics
→ VictoriaLogs for logs
→ optional trace backend
→ Grafana

PKE LLM observations
→ Langfuse

Scope:
- Add OpenTelemetry instrumentation for the reasoning workflow
- Add spans for:
  - load job
  - load job analysis
  - build candidate evidence pack
  - canonical hydration
  - objective scoring
  - build LLM payload
  - Ollama inference
  - schema validation
  - persistence
- Add metrics:
  - command duration
  - LLM duration
  - prompt tokens
  - completion tokens
  - candidate evidence count
  - candidate pack size
  - evidence count sent per requirement
  - reasoning failures
  - validation failures
- Add structured logs with trace correlation
- Add Langfuse tracing for LLM calls
- Correlate telemetry using:
  - traceId
  - runIdentity
  - jobDescriptionId
  - jobAnalysisId
  - candidatePackHash
- Add Docker Compose profiles:
  - observability-lite
  - observability-full
- observability-lite includes:
  - OpenTelemetry Collector
  - VictoriaMetrics
  - VictoriaLogs
  - Grafana
- observability-full additionally includes:
  - Langfuse and required dependencies
- Add provisioned Grafana data sources and dashboards
- Add a reasoning performance dashboard
- Add README instructions for starting and testing the local stack
- Keep all observability integrations optional
- Preserve console logging when observability is disabled

Dashboard panels:
- jobs reason total duration
- Ollama inference duration
- prompt and completion tokens
- candidate-pack size
- candidates per requirement
- evidence sent to reasoner
- schema-validation failures
- reasoning failures
- latency by model and prompt version

Rules:
- Application code depends only on observability ports and OpenTelemetry APIs
- Grafana, VictoriaMetrics, VictoriaLogs and Langfuse must not be imported into domain or application modules
- High-cardinality IDs must not be metric labels
- Prompt and evidence contents must not be emitted as ordinary logs
- Sensitive professional content must be opt-in for Langfuse tracing
- Telemetry failures must not fail the main workflow
- Local observability must not be required to run the CLI
- Use configurable sampling and retention
- Do not run Loki and VictoriaLogs simultaneously in the default profile

Out of scope:
- production deployment
- alert paging
- cloud observability vendors
- LLM benchmarking
- GPU-level telemetry
- distributed microservices

Acceptance criteria:
- a jobs reason execution produces correlated traces, metrics and logs
- Ollama inference time is distinguishable from total command time
- token counts and candidate-pack size are visible
- Grafana dashboards work locally
- Langfuse shows the corresponding LLM generation
- disabling observability does not break the CLI
- telemetry backend failure does not fail reasoning
