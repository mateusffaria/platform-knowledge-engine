Add an evaluation framework for evidence retrieval and reasoning.

Context:
The Professional Knowledge Engine currently supports:

- canonical professional knowledge
- reconciliation and trust policies
- semantic and structured retrieval
- PKQL and metadata-driven planning
- job ingestion and analysis
- candidate evidence enrichment
- LLM-based evidence reasoning
- local application and AI observability

The system lacks a reproducible way to detect quality regressions when retrieval policies, prompts, models or reasoning logic change.

Goal:
Introduce a deterministic evaluation framework for retrieval, Candidate Evidence Packs and Curated Evidence Packs.

Architecture:

Evaluation Dataset
→ Evaluation Runner
→ Pipeline Execution
→ Deterministic Assertions
→ Evaluation Report
→ Metrics and Langfuse observations

Scope:
- Create an evaluation module using hexagonal boundaries
- Add EvaluationScenario
- Add EvaluationDataset
- Add EvaluationExpectation
- Add EvaluationRun
- Add EvaluationResult
- Add fixture-based golden scenarios
- Add CLI commands:
  - pke eval run
  - pke eval run <scenario-id>
  - pke eval list
  - pke eval show <run-id>
- Evaluate independently:
  - retrieval results
  - candidate evidence association
  - reasoning output
- Support deterministic expectations:
  - expected evidence IDs
  - forbidden evidence IDs
  - expected minimum/maximum coverage status
  - expected missing requirements
  - maximum evidence count
  - required provenance
  - no fabricated evidence
  - deterministic schema validity
- Add aggregate metrics:
  - evidence precision at K
  - evidence recall at K
  - requirement coverage accuracy
  - missing-evidence accuracy
  - unsupported-selection rate
  - provenance completeness
  - schema-validation success rate
  - average reasoning latency
  - prompt and completion tokens
- Export reports as:
  - concise CLI output
  - JSON
  - Markdown
- Emit evaluation metrics through existing observability contracts
- Attach evaluation metadata to Langfuse runs when enabled
- Record:
  - model
  - provider
  - promptVersion
  - candidatePackVersion
  - datasetVersion
  - git SHA
- Add documentation and update the roadmap

Initial golden scenarios:
1. Exact technology coverage:
   - TypeScript
   - Node.js
   - PostgreSQL

2. Conceptual leadership coverage:
   - select leadership-related evidence
   - reject unrelated performance-only evidence

3. Missing technology:
   - Kubernetes or Flutter must remain missing when unsupported

4. Empty Candidate Evidence Pack:
   - all requirements missing
   - no fabricated evidence

5. Redundant candidates:
   - select the strongest complementary evidence
   - reject duplicate or weaker alternatives

6. Trust policy:
   - rejected, superseded and needs_review claims never appear

Rules:
- Golden expectations are versioned test data
- Evaluation must not modify canonical knowledge
- Evaluation must not automatically promote claims
- Deterministic assertions are authoritative in this milestone
- LLM-as-judge is not required in the initial implementation
- A selected evidence ID must exist in the supplied Candidate Evidence Pack
- Missing evidence must not be penalized when the canonical store truly lacks it
- Quality metrics must be separated from performance metrics
- Evaluation backend failures must not affect normal CLI workflows

Out of scope:
- resume generation
- cover-letter generation
- model leaderboard
- automated provider comparison
- fine-tuning
- subjective writing-quality evaluation
- production alerting

Acceptance criteria:
- pke eval run executes all golden scenarios
- reports identify passed and failed expectations
- fabricated evidence is detected
- expected missing requirements are validated
- retrieval and reasoning regressions can be distinguished
- results contain model, prompt, dataset and git versions
- evaluation metrics appear in the local observability stack
- existing workflows remain functional
