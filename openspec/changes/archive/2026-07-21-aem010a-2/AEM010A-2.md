Refine compound job requirements and evidence coverage granularity.

Context:
The Evidence Reasoner is functioning correctly and safely abstains when evidence is missing.

A real run produced:
- 55 retrieved candidates
- 16 candidates supplied to the reasoner
- 1 selected evidence item
- 4 missing requirements

The requirement "Strong knowledge of Go and PostgreSQL" was evaluated as one combined requirement, even though PostgreSQL evidence exists and Go evidence does not.

This reduces explainability and limits downstream resume planning.

Goal:
Decompose compound requirements into atomic requirement components so each skill, technology or capability can be evaluated independently.

Scope:
- Add RequirementComponent or AtomicJobRequirement
- Decompose compound requirements such as:
  - "Go and PostgreSQL"
  - "Terraform and AWS"
  - "Docker and Kubernetes"
- Preserve the parent JobRequirement
- Add component-level identifiers and provenance
- Evaluate evidence coverage per component
- Aggregate component coverage into parent coverage deterministically
- Add aggregation rules:
  - all strong/partial → parent strong or partial
  - mixed covered and missing → parent partial
  - all missing → parent missing
- Update retrieval intent generation to search per component
- Associate candidate evidence with requirement components
- Update Candidate Evidence Pack
- Update Curated Evidence Pack
- Update Resume Content Planner to consume component-level coverage
- Preserve backward compatibility for existing persisted runs
- Deduplicate warnings by code and message
- Add diagnostics for:
  - parent requirement count
  - atomic component count
  - selected evidence per component
- Add tests for:
  - Go and PostgreSQL
  - Terraform and AWS
  - Docker and Kubernetes
  - non-compound conceptual requirements
  - mixed covered and missing components

Rules:
- Do not split ordinary phrases indiscriminately
- Split only when the parser identifies coordinated independently testable requirements
- Preserve original requirement text
- Missing one component must not erase evidence for another
- Parent coverage must be derived from component coverage
- The LLM must not invent components not supported by the source text
- Warning aggregation must be deterministic and deduplicated

Acceptance criteria:
- PostgreSQL can be covered independently from Go
- AWS can be evaluated independently from Terraform
- Kubernetes can remain missing while Docker is covered
- Parent requirements remain traceable to the original job source
- Resume planning can use covered components without overstating the parent requirement
- warnings are not duplicated
