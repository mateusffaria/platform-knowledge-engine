Extract trusted-knowledge assessment and conflict resolution into a dedicated reconciliation module.

Context:
The project currently supports claim statuses, conflict detection and a human review workflow. These responsibilities are currently located inside existing knowledge or ingestion modules, but reconciliation has emerged as a distinct business capability.

Goal:
Introduce a dedicated reconciliation module responsible for evaluating claims from multiple sources, detecting conflicts, managing claim assessment status and controlling eligibility for indexing and retrieval.

Architecture:
Keep the project as a modular monolith with hexagonal boundaries inside each module.

Target modules:
- ingestion: parse and normalize source documents
- knowledge: own and persist EvidenceClaims, KnowledgeAssets and source provenance
- reconciliation: assess claims, detect conflicts and manage review decisions
- retrieval: index and retrieve eligible knowledge

Scope:
- Create src/modules/reconciliation
- Move conflict detection rules into reconciliation
- Move claim assessment and review use cases into reconciliation
- Move CLI commands for claims review, confirmation and rejection into reconciliation interfaces
- Introduce explicit reconciliation domain concepts:
  - ClaimAssessment
  - ClaimStatus
  - Conflict
  - ConflictSeverity
  - ReconciliationResult
- Keep EvidenceClaim owned by the knowledge module
- Introduce ports for reading claims and persisting assessments
- Ensure reconciliation does not import knowledge infrastructure directly
- Preserve existing CLI behavior
- Preserve existing database data and migrations where possible
- Update dependency injection/composition root
- Update architecture and trusted-knowledge documentation
- Add tests for module boundaries and reconciliation behavior

Domain rules:
- Absence of a claim in another source is not a conflict
- Compatible claims may corroborate each other
- Incompatible facts about the same normalized subject require review
- User confirmation overrides automated assessment
- Rejected and superseded claims remain auditable but inactive
- Reconciliation controls indexing eligibility, but retrieval owns vector indexing and search
- Knowledge owns EvidenceClaim identity and provenance

Dependency rules:
- reconciliation application may depend on reconciliation domain and explicit ports
- reconciliation must not import repositories, Drizzle schemas or pgvector adapters from other modules
- cross-module communication must happen through application contracts
- CLI commands may call reconciliation use cases only

Out of scope:
- LLM-based conflict resolution
- hybrid retrieval
- document generation
- web UI
- new source parsers
- redesigning the entire database schema

Acceptance criteria:
- Existing claims review/confirm/reject commands continue working
- Conflict detection is owned by reconciliation
- EvidenceClaim remains owned by knowledge
- Rejected and superseded claims remain excluded from indexing
- Existing tests pass
- New module-boundary and reconciliation tests pass
- Documentation clearly explains module responsibilities
