Add hybrid retrieval and evidence pack generation.

Context:
The project currently supports:
- professional source ingestion
- structured knowledge persistence in PostgreSQL
- semantic indexing and search using pgvector
- Ollama embeddings
- trusted knowledge assessment
- claim reconciliation
- indexing eligibility based on claim status

Goal:
Combine structured retrieval and semantic retrieval into a single application-level capability that produces a ranked, deduplicated and traceable Evidence Pack.

Architecture:
Keep the project as a modular monolith with hexagonal boundaries.

The retrieval module owns:
- query planning
- structured retrieval
- semantic retrieval orchestration
- result merging
- deduplication
- ranking
- Evidence Pack generation

The retrieval module must not own:
- claim reconciliation
- truth resolution
- source ingestion
- document generation

Scope:
- Introduce a QueryPlanner application service or domain service
- Add StructuredKnowledgeSearch port
- Reuse the existing semantic VectorStore port
- Add HybridSearch use case
- Add EvidencePack domain model
- Add EvidenceItem domain model
- Merge structured and semantic results
- Deduplicate results by EvidenceClaim or KnowledgeAsset identity
- Preserve provenance and trust metadata
- Add ranking policy
- Add CLI command:
  - pke retrieve "<query>"
- Add options:
  - --limit
  - --min-score
  - --claim-status
  - --subject-type
  - --json
  - --verbose
- Add tests for:
  - structured-only matches
  - semantic-only matches
  - duplicate results returned by both strategies
  - rejected and superseded claims excluded
  - deterministic ordering
  - provenance preservation
- Update architecture, roadmap and retrieval documentation

Evidence Pack must include:
- original query
- retrieval strategies used
- ranked evidence items
- claim status
- confidence score
- semantic similarity score when available
- structured match score when available
- final ranking score
- source references
- source excerpts
- retrieval timestamp
- warnings or unresolved limitations

Initial query planning rules:
- Exact skill, company, role, technology and date-like terms should trigger structured retrieval
- Natural-language intent and conceptual queries should trigger semantic retrieval
- Mixed queries should use both
- The initial planner must be deterministic and rule-based
- Do not use an LLM for query planning in this milestone

Ranking rules:
- Ranking must be deterministic
- Confirmed claims should rank above equivalent single-source claims
- Rejected, superseded and needs-review claims must not appear
- Exact structured matches should receive a defined boost
- Semantic similarity may contribute to the final score
- Ranking weights must be configurable and documented
- Do not present the final score as an objective probability of truth

Module boundaries:
- Retrieval may consume eligible claims through explicit knowledge/reconciliation application contracts
- Retrieval must not import knowledge or reconciliation infrastructure directly
- CLI must call retrieval use cases only
- Postgres and pgvector implementations remain infrastructure adapters

Out of scope:
- LLM-based query planner
- resume generation
- job description parsing
- agent orchestration
- reranking with an LLM or cross-encoder
- benchmarking
- UI/dashboard

Acceptance criteria:
- pke retrieve "<query>" returns an Evidence Pack
- Exact and semantic matches can be combined
- Duplicate claims appear only once
- Each result remains traceable to its original source
- Ineligible claims are excluded
- Ranking is deterministic and tested
- Existing pke search behavior remains functional
- Existing tests continue passing


Suggested Innitial Module:

```typescript
interface EvidencePack {
  query: string;
  strategies: Array<"structured" | "semantic">;
  items: EvidenceItem[];
  generatedAt: Date;
  warnings: string[];
}

interface EvidenceItem {
  evidenceClaimId?: string;
  knowledgeAssetId: string;
  claimType?: string;
  claimText: string;
  claimStatus: "confirmed" | "single_source";
  confidenceScore: number;
  semanticScore?: number;
  structuredScore?: number;
  finalScore: number;
  sources: SourceReference[];
}
```
