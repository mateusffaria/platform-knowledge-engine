Refactor the Query Planner to be fully metadata-driven.

Context:
The project now supports:
- an evolved Canonical Career Model
- PKQL parsing into a Query AST
- structured and semantic retrieval
- explicit structured filters as candidate-set constraints
- metadata persisted in the Professional Knowledge Store

The Query Planner must no longer own hardcoded domain vocabulary or language-specific classification heuristics.

Goal:
Make query planning depend only on:
- the parsed PKQL AST
- normalized metadata matches
- available retrieval capabilities

Architecture:
Raw query
→ PKQL Parser
→ Query AST
→ Metadata Matcher
→ Query Planner
→ Retrieval Plan

Scope:
- Remove hardcoded natural-language term lists from QueryPlanner
- Remove hardcoded technology, role, company, project and skill vocabularies
- Remove direct iteration over internal KnowledgeMetadata groups from QueryPlanner
- Introduce or refine a MetadataMatcher port
- Make the metadata provider return normalized MetadataMatch objects
- Add MetadataCategory:
  - skill
  - technology
  - organization
  - role
  - project
  - product
  - initiative
- Support match types:
  - exact
  - prefix
  - partial
  - alias
- Make QueryPlanner consume Query AST and MetadataMatch[] only
- Keep semantic retrieval as the default fallback
- Use structured retrieval when:
  - PKQL contains explicit filters
  - persisted metadata matches are found
- Use both structured and semantic retrieval when:
  - explicit filters coexist with semantic text
  - metadata matches coexist with conceptual free text
- Keep explicit structured filters as candidate-set constraints
- Add tests proving new technologies, organizations and roles require no planner code changes
- Add tests for English and Portuguese queries
- Add tests for queries with no metadata matches
- Add tests for mixed PKQL and free-text queries
- Update retrieval and PKQL documentation

Rules:
- User vocabulary must come from persisted knowledge metadata
- QueryPlanner must not inspect raw database metadata structures
- QueryPlanner must not contain language stopword lists
- Internal metadata category names may remain canonical English identifiers
- Language localization and aliases belong to parsing or metadata matching adapters
- RetrievalStrategy remains an explicit application capability
- EvidenceClaimStatus remains an explicit domain state machine
- Semantic retrieval is the fallback strategy

Out of scope:
- LLM-based planning
- fuzzy embedding-based metadata matching
- query rewriting
- reranking
- document generation
- agent orchestration

Acceptance criteria:
- QueryPlanner contains no hardcoded professional-domain vocabulary
- QueryPlanner contains no language-specific natural-language term list
- New metadata values become discoverable without code changes
- PKQL explicit filters continue working
- Portuguese and English queries produce equivalent normalized matches where applicable
- Existing retrieval behavior remains compatible
- Existing tests continue passing
