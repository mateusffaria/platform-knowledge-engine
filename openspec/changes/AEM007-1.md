Add a bounded LLM Evidence Reasoner over canonical Candidate Evidence Packs.

Context:
The evidence-reasoning module can now produce Candidate Evidence Packs by:

- hydrating retrieval results from the canonical knowledge store
- traversing canonical relationships
- grouping evidence by job requirement
- calculating deterministic objective signals
- preserving trust and provenance

The system now needs qualitative reasoning to select complementary evidence, evaluate requirement coverage and explain limitations.

Goal:
Introduce a bounded EvidenceReasoner that evaluates preselected canonical evidence without accessing the database, vector store or unrestricted retrieval capabilities.

Architecture:

Candidate Evidence Pack
→ Evidence Reasoner
→ Requirement Coverage Analysis
→ Cross-requirement Deduplication
→ Curated Evidence Pack

Scope:

- Add EvidenceReasoner port
- Add LlmEvidenceReasoner implementation
- Reuse the existing LlmProvider port
- Add CuratedEvidencePack
- Add RequirementCoverage
- Add EvidenceSelection
- Add EvidenceRejection
- Add CoverageStatus:
  - strong
  - partial
  - weak
  - missing
- Add CLI command:
  - pke jobs reason <job-id>
- Add options:
  - --model
  - --json
  - --verbose
- Use structured output validated through Zod or equivalent
- Add prompt versioning
- Add Langfuse tracing through existing observability contracts
- Add requirement-by-requirement reasoning
- Add cross-requirement deduplication
- Add warnings and limitations
- Add deterministic conversion from qualitative coverage to an optional numeric display score
- Persist reasoning runs with:
  - jobDescriptionId
  - jobAnalysisId
  - candidatePack version or hash
  - provider
  - model
  - promptVersion
  - createdAt
- Add tests using mocked LlmProvider outputs
- Update documentation and ADRs

CuratedEvidencePack should include:

- jobDescriptionId
- jobAnalysisId
- provider
- model
- promptVersion
- createdAt
- overall coverage summary
- requirement coverage entries
- recommended evidence
- discarded evidence
- missing evidence
- warnings
- limitations

RequirementCoverage should include:

- requirementId
- requirement text
- importance
- coverageStatus
- selectedEvidenceIds
- rejectedCandidateEvidenceIds
- strengthFactors
- limitations
- explanation

EvidenceSelection should include:

- evidenceClaimId
- reason for selection
- contribution to the requirement
- complementaryEvidenceIds when applicable
- exaggerationRisk:
  - low
  - medium
  - high

EvidenceRejection should include:

- evidenceClaimId
- reason:
  - irrelevant
  - weak
  - redundant
  - unsupported_scope
  - lower_quality_alternative
  - insufficient_provenance

Rules:

- The reasoner receives only Candidate Evidence Packs
- The reasoner must not access repositories, PostgreSQL, pgvector or external search
- The reasoner must not create new professional evidence
- The reasoner must not modify trust status
- The reasoner must not replace canonical claim content
- The reasoner may select, reject, compare and explain evidence
- The reasoner must prefer omission over unsupported interpretation
- Objective signals must be provided to the model, but the model must not rewrite them
- Coverage is a qualitative assessment, not proof that the candidate satisfies a hiring requirement
- A requirement with no eligible evidence must be marked missing
- A skill-only claim without contextual usage should generally not produce strong coverage
- Multiple redundant claims should not all be selected
- A measurable, contextual and trusted claim should generally rank above an isolated keyword match
- LLM output must be schema-validated
- Invalid output must fail safely without modifying the Candidate Evidence Pack

Reasoning guidance:

The model should assess:

- direct relevance
- contextual depth
- measurable impact
- demonstrated ownership
- seniority signal
- trust and provenance
- complementarity
- redundancy
- exaggeration risk
- missing scope

The model should not merely reorder by semantic similarity.

Example:

Requirement:
"Technical leadership across teams"

Evidence A:
"Mentored engineers on architectural decisions."

Evidence B:
"Acted as technical point of contact for leadership, surfacing risks and trade-offs."

Evidence C:
"Improved API latency."

Expected:
- Select A and B as complementary evidence
- Reject C as not directly demonstrating leadership
- Mark coverage partial if organizational scope is not explicit
- Explain the limitation

Out of scope:

- resume writing
- cover-letter writing
- interview-answer generation
- autonomous agent loops
- unrestricted tools
- multiple reasoner agents
- LLM benchmarking
- fine-tuning

Acceptance criteria:

- pke jobs reason <job-id> produces a validated Curated Evidence Pack
- Evidence is evaluated requirement by requirement
- Redundant evidence is discarded
- Missing requirements remain explicit
- Strong, partial, weak and missing coverage are distinguishable
- Every selected item references canonical evidence
- The model cannot create new evidence
- Invalid model output fails safely
- Provider, model and prompt version are traceable
- LLM calls are observable
- Existing deterministic workflows continue passing
